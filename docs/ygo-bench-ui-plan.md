# YGO Bench Proper UI Plan

## Goal

Build a proper browser UI for YGO Bench that makes benchmark runs inspectable, comparable, and replayable without weakening the trace-first architecture.

The UI should answer:

> What happened in this duel, what did the agent see, what did it choose, why did it choose it, and how did that affect benchmark score?

The UI is not a duel client. It is a benchmark run explorer and replay tool backed by existing run artifacts.

## Product Scope

The first proper UI should support three workflows:

1. Browse benchmark runs.
2. Replay one run frame by frame.
3. Compare agents across a suite.

The source of truth remains:

- `trace.jsonl`
- `final-score.json`
- `metadata.json`
- `model-transcript.md`
- `reduced-state.json`
- `mvp-summary.json`
- `mvp-summary.csv`

The UI should never talk directly to `ocgcore-wasm` for board state. It consumes traces and summaries generated by the benchmark runner. That keeps the viewer honest: if the UI shows it, the scoring system saw it too.

```txt
      styles/

```

Build outputs:

```txt
apps/ygo-bench/dist-viewer/
```

CLI integration:

```sh
pnpm bench ui
pnpm bench ui --port 4173
pnpm bench ui --run benchmark-runs/real-run-...
pnpm bench ui --summary benchmark-runs/mvp-summary.json
```

The existing `serve-trace` command can remain as a low-level trace streamer, but `ui` should become the normal user-facing command.

## Server Shape

Use a small Node HTTP server in the CLI package.

Routes:

```txt
GET /                         -> viewer app
GET /assets/...               -> viewer static assets
GET /api/runs                 -> discovered run directories
GET /api/runs/:id             -> score, metadata, reduced state, artifact paths
GET /api/runs/:id/trace       -> trace JSON array or JSONL stream
GET /api/runs/:id/transcript  -> model transcript markdown
GET /api/summaries            -> available suite summaries
GET /api/summaries/:id        -> summary JSON
WS  /api/runs/:id/live        -> appended trace frames for live runs
```

Run discovery should scan `apps/ygo-bench/benchmark-runs/` for directories containing `trace.jsonl` and `final-score.json`.

The API should be local-only by default:

```txt
host: 127.0.0.1
```

## UI Layout

Use an operational-tool layout, not a marketing page.

```txt
┌──────────────────────────────────────────────────────────────┐
│ Header: suite/run selector, agent, scenario, score, status   │
├───────────────┬───────────────────────────────┬──────────────┤
│ Run List      │ Board Replay                   │ Timeline     │
│ Summary       │                               │ Events       │
│ Filters       │                               │ Decisions    │
├───────────────┴───────────────────────────────┴──────────────┤
│ Decision / Prompt / Legal Actions / Model Reasoning           │
└──────────────────────────────────────────────────────────────┘
```

Primary route:

```txt
/runs/:runId
```

Secondary route:

```txt
/suites/:summaryId
```

No landing page. The first screen should show recent runs and the latest suite summary.

## Core Screens

### 1. Run Browser

Purpose: quickly find and open benchmark runs.

Shows:

- scenario id
- family
- agent
- score
- win/loss
- decisions
- illegal actions
- invalid JSON
- model errors
- token count
- latency
- created time

Filters:

- agent
- scenario family
- win/loss
- score range
- model errors
- latest suite only

Acceptance criteria:

- Can open any run generated by `pnpm bench run`.
- Can open viewer links generated by `pnpm bench eval`.
- Clearly flags runs with model errors or illegal action IDs.

### 2. Suite Summary

Purpose: prove benchmark value at a glance.

Shows:

- aggregate table by agent
- weighted objective score
- win rate
- average decisions
- average LP delta
- model error rate
- average token count
- per-family breakdown

Charts:

- score by agent
- score by family
- scenario heatmap: scenario x agent

Acceptance criteria:

- The latest `mvp-summary.json` renders without manual conversion.
- Random, greedy, and LLM score separation is visually obvious.
- Each row links to the corresponding run replay.

### 3. Replay View

Purpose: watch one duel unfold.

Shows:

- turn / phase
- LP totals
- player 0 and player 1 field
- hand count
- own hand when the observation belongs to that player
- deck count
- extra deck count
- graveyard
- banished
- timeline
- current prompt
- legal actions
- chosen action
- model reason
- line quality

Controls:

- previous frame
- next frame
- previous decision
- next decision
- play/pause
- speed selector
- jump to turn
- jump to phase

Acceptance criteria:

- Can replay a completed `trace.jsonl`.
- Can show the board at every decision frame.
- Decision frames show exactly what the model saw.
- Hidden engine response payloads are not displayed.

### 4. Decision Panel

Purpose: inspect the agent boundary.

Shows:

- prompt type
- active player
- observation JSON
- legal action list
- selected action
- model reason
- token count
- model error if any
- fallback reason if fallback occurred

Acceptance criteria:

- Legal actions shown in UI match `trace.jsonl`.
- Engine response bytes/objects are not exposed.
- Invalid JSON and illegal action counts are visible from score data.

### 5. Transcript Drawer

Purpose: inspect full model prompt/response history.

Shows:

- rendered `model-transcript.md`
- decision anchors
- copyable prompt JSON
- copyable response JSON

Acceptance criteria:

- Opens from any decision frame.
- Highlights the current decision.
- Does not leak hidden engine response payloads.

### 6. Metadata Panel

Purpose: make runs reproducible.

Shows:

- ocgcore-wasm version
- ocgcore version
- card data hash
- card scripts commit
- BabelCDB commit
- LFLists commit/path when available
- banlist hash
- scenario hash
- max decisions
- created time

Acceptance criteria:

- Renders `metadata.json`.
- Clearly marks missing optional upstream commits as unavailable, not as success.

## Data Model

Normalize loaded artifacts into UI models.

```ts
interface RunIndexItem {
  id: string;
  path: string;
  scenarioId: string;
  agentId: string;
  family: string;
  score: number;
  won: boolean;
  decisions: number;
  modelErrors: number;
  invalidJson: number;
  illegalActions: number;
  createdAt?: string;
}
```

```ts
interface ReplayFrame {
  index: number;
  kind: "event" | "decision" | "engine" | "error";
  turn?: number;
  phase?: string;
  text?: string;
  event?: string;
  state?: RealReducedState;
  decision?: DecisionPayload;
}
```

The UI should tolerate old traces that lack newer fields. If a field is missing, show `Unknown` or hide that detail rather than crashing.

## Replay State

The replay reducer should use decision snapshots already stored in trace frames:

```txt
trace decision frame
  -> reducedState snapshot
  -> board view
```

For non-decision event frames, use the nearest previous snapshot for MVP. Later, we can reconstruct state per event with the reducer.

Acceptance criteria:

- Decision frame board state is correct.
- Non-decision frames do not show obviously impossible board states.
- UI does not mutate trace data.

## Visual Direction

The UI should feel like a dense benchmark/debugging tool.

Style:

- restrained dark or neutral theme
- compact cards
- clear zone boundaries
- colored event categories
- minimal animation
- high information density

Avoid:

- landing-page hero sections
- decorative gradients/orbs
- oversized marketing cards
- card-art dependency in MVP

Card tiles should show:

- card name
- code
- ATK when available
- zone
- face-down state
- revealed/hidden marker

Face-down cards:

```txt
Set card
Hidden
```

Do not show hidden card names in model-observation views.

## Live Mode

Live mode should build on the existing trace stream idea.

Runner behavior:

```txt
pnpm bench run scenarios/real/lethal-001.json --agent llm --viewer
```

Future behavior:

```txt
pnpm bench run scenarios/real/lethal-001.json --agent llm --ui
```

The CLI can:

1. create the run directory
2. start the UI server
3. stream appended `trace.jsonl` frames over WebSocket
4. print the local URL

Acceptance criteria:

- UI can open before the run finishes.
- Timeline updates as trace lines are appended.
- Completed runs can be replayed without the runner still active.

## Implementation Milestones

### Milestone 1: Viewer App Scaffold

Deliverable:

```sh
pnpm --filter @ygo-bench/app viewer:build
pnpm bench ui
```

Tasks:

- Add Vite React app under `src/viewer-app`.
- Add `viewer:build` script.
- Add CLI `ui` command.
- Serve static viewer assets from local HTTP server.
- Add `/health`.

Acceptance criteria:

- Browser opens or URL prints.
- App loads without trace data.
- App shows empty state and available run count.

### Milestone 2: Run Discovery API

Deliverable:

```sh
pnpm bench ui
GET /api/runs
```

Tasks:

- Scan `benchmark-runs`.
- Parse `final-score.json`.
- Parse `metadata.json` when present.
- Sort newest first.
- Return stable run ids.

Acceptance criteria:

- UI lists recent runs.
- Broken/incomplete runs are skipped or marked clearly.
- Tests cover run discovery with fixture directories.

### Milestone 3: Suite Summary Page

Deliverable:

```txt
/suites/mvp-summary
```

Tasks:

- Load `mvp-summary.json`.
- Render aggregate table.
- Render scenario x agent table.
- Link records to run pages.

Acceptance criteria:

- Latest MVP eval is inspectable.
- Agent score separation is visible.
- Viewer links resolve locally.

### Milestone 4: Run Replay Page

Deliverable:

```txt
/runs/:runId
```

Tasks:

- Load trace.
- Load score.
- Load metadata.
- Build replay frame list.
- Render board from decision snapshots.
- Render timeline.
- Add decision navigation.

Acceptance criteria:

- Can replay a real OpenAI run.
- Can replay a random/greedy/oracle run.
- Board, timeline, prompt, actions, and chosen action stay in sync.

### Milestone 5: Model Boundary Panels

Deliverable:

```txt
Decision panel + transcript drawer
```

Tasks:

- Render observation JSON.
- Render legal action list.
- Render chosen action and reason.
- Render token count and model errors.
- Render transcript markdown.
- Add hidden-response leak check in tests.

Acceptance criteria:

- The UI never renders hidden `response` payloads.
- Prompt and legal action views match trace frame data.
- Transcript opens at the selected decision.

### Milestone 6: Live Trace Mode

Deliverable:

```sh
pnpm bench ui --run <active-run>
```

Tasks:

- Add WebSocket subscription per run.
- Stream appended trace lines.
- Merge live frames into replay state.
- Show run status.

Acceptance criteria:

- UI updates while a run is still writing `trace.jsonl`.
- Refreshing the page replays existing frames.
- WebSocket disconnect does not lose loaded data.

### Milestone 7: Polish And Verification

Tasks:

- Add keyboard shortcuts for replay navigation.
- Add responsive layout.
- Add empty/error/loading states.
- Add visual markers for hidden cards.
- Add score badges.
- Add test fixtures based on real traces.

Acceptance criteria:

- `pnpm --filter @ygo-bench/app viewer:build` passes.
- Unit tests pass.
- Manual check on latest `mvp-summary.json` passes.
- Manual check on a real OpenAI run passes.

## Testing Strategy

Add tests for:

- run discovery
- summary loading
- trace parsing
- replay frame selection
- hidden response omission
- hidden card display
- route/API behavior

Use small fixture traces in:

```txt
apps/ygo-bench/src/viewer-app/__fixtures__/
```

Do not make tests depend on the current large `benchmark-runs` directory.

## Out Of Scope For First UI

- Card art rendering.
- Deck editor.
- Scenario editor.
- Direct ocgcore control from browser.
- Authentication.
- Hosted deployment.
- EDOPro replay export.
- Mobile-first design.
- Full event-by-event reducer replay for every frame.

## Definition Of Done

The proper UI is done when:

1. `pnpm bench ui` starts a local browser UI.
2. The UI lists benchmark runs.
3. The UI opens a suite summary and shows agent comparison.
4. The UI opens a run and replays `trace.jsonl`.
5. Board, LP, turn, phase, graveyard, banished, hand count, deck count, and extra deck count are visible.
6. Decision frames show prompt, legal actions, chosen action, and model reason.
7. Model transcript and metadata are inspectable.
8. Hidden card names and hidden engine responses do not leak in model-observation views.
9. Live mode can stream appended trace frames.
10. Build and tests pass.

Recommended final verification:

```sh
pnpm --filter @ygo-bench/app build
pnpm --filter @ygo-bench/app viewer:build
pnpm --filter @ygo-bench/app test
pnpm bench eval suites/mvp.json --agents random,greedy,llm --runs 1 --model gpt-4o-mini --viewer
pnpm bench ui
```
